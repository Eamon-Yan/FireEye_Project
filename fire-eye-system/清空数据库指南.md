# 🔥 FireEye 数据库清空指南

## 为什么需要清空数据库？

当你修改了后端的节点分类逻辑后，已经存在的节点不会自动更新类型。需要清空数据库并重新导入文档，才能看到新的分类效果。

---

## 方法一：使用 Docker 方式清空（推荐）⭐

说明：本方法只清空图数据库中的业务节点和关系，不会删除 Neo4j 数据卷，也不会重置 Neo4j 登录密码。

### 步骤：

1. **确保 Docker 后端和数据库服务正在运行**
   ```bash
   cd fire-eye-system
   docker compose up -d neo4j redis backend
   ```

2. **在后端容器中执行清空脚本**
   ```bash
   docker exec -it fire-eye-backend python clear_database.py
   ```

3. **确认清空操作**
   - 脚本会提示你输入 `yes` 确认
   - 输入 `yes` 后按回车

4. **查看清空结果**
   - 脚本会显示清空前后的统计信息
   - 确认节点数和关系数都变为 0

5. **重新上传测试文档**
   - 打开浏览器访问 http://localhost:8080/upload
   - 上传 `测试文档-火灾调查报告示例.txt`
   - 等待处理完成

6. **查看图谱验证**
   - 访问 http://localhost:8080/graph
   - 检查节点颜色是否有变化：
     - 🔴 红色 (FireEvent) - 火灾事件
     - 🟠 橙色 (Hazard) - 安全隐患
     - 🟣 紫色 (Consequence) - 后果影响

## 方法二：使用容器内数据库管理工具

### 步骤：

1. **进入项目目录**
   ```bash
   cd fire-eye-system
   ```

2. **在后端容器中运行重置命令**
   ```bash
   docker exec -it fire-eye-backend python scripts/db_manager.py reset
   ```

3. **确认操作**
   - 输入 `y` 确认重置

## 方法三：使用Neo4j浏览器（手动）

### 步骤：

1. **打开Neo4j浏览器**
   - 访问 http://localhost:7474
   - 使用用户名 `neo4j` 和密码 `password` 登录

2. **执行清空命令**
   ```cypher
   MATCH (n) DETACH DELETE n
   ```

3. **验证清空**
   ```cypher
   MATCH (n) RETURN count(n) as node_count
   ```
   - 应该返回 `node_count: 0`

## 验证新的节点分类逻辑

清空数据库后，重新上传测试文档，你应该看到：

### 预期的节点类型分布：

根据测试文档内容，应该有以下三种类型的节点：

1. **🔴 FireEvent (火灾事件)** - 红色
   - 示例：电路短路、产生电弧、引燃包装材料、火势蔓延

2. **🟠 Hazard (安全隐患)** - 橙色
   - 示例：电线绝缘层破损、电气线路老化、安全检查不到位

3. **🟣 Consequence (后果影响)** - 紫色
   - 示例：造成财产损失、人员伤亡、设备损坏

### 如何验证：

1. 在图谱页面查看节点颜色
2. 点击不同颜色的节点查看详情
3. 检查"类型"字段是否正确显示为：火灾事件/安全隐患/后果影响

## 常见问题

### Q1: 清空后前端还显示旧数据？
**A:** 刷新浏览器页面（Ctrl+F5 强制刷新）

### Q2: 清空脚本报错连接失败？
**A:** 确保 Docker 中的 backend、Neo4j、Redis 都在运行：
```bash
docker ps
```
必要时先执行：
```bash
cd fire-eye-system
docker compose up -d neo4j redis backend
```

### Q3: 重新上传后节点颜色还是一样？
**A:** 可能是以下原因：
- 后端镜像中的代码未更新
- 后端容器没有重启
- 浏览器缓存问题（清除缓存或使用无痕模式）

### Q4: 如何重启后端服务？
**A:**
```bash
cd fire-eye-system
docker compose restart backend
```

### Q5: 什么时候应该使用 `docker compose down -v`？
**A:** 只在以下场景才使用：
- Neo4j 因旧数据卷和旧容器初始化参数冲突而反复重启
- 你明确要彻底重置 Neo4j 数据和认证状态

执行方式：
```bash
cd fire-eye-system
docker compose down -v
docker compose -f docker-compose.yml -f docker-compose.init.yml up -d
```

注意：
- 这会删除 Neo4j 持久化卷
- 会丢失 Neo4j 中已有数据
- 重新启动时会用初始账号密码 `neo4j/password` 重新初始化

## 注意事项

⚠️ **重要提醒：**
- 清空数据库会删除所有节点和关系，无法恢复
- 建议在测试环境中操作
- 如果有重要数据，请先备份
- `python clear_database.py` 和 `python scripts/db_manager.py reset` 只清空业务数据，不会删除 Neo4j 数据卷
- 只有 `docker compose down -v` 才会删除 Neo4j 数据卷并触发首次初始化流程

💡 **提示：**
- 清空操作很快，通常只需要几秒钟
- 重新导入文档需要等待LLM处理，可能需要10-30秒
- 如果多次测试，建议保存测试文档以便重复使用

## 技术说明

### 节点分类逻辑

后端使用关键词匹配来智能分类节点：

- **Hazard (隐患)**: 包含"老化"、"破损"、"故障"、"缺陷"等关键词
- **Consequence (后果)**: 包含"造成"、"导致"、"伤亡"、"损失"等关键词  
- **FireEvent (火灾事件)**: 包含"起火"、"燃烧"、"爆炸"、"烟雾"等关键词

详见：`fire-eye-system/backend/app/db/neo4j_manager.py` 中的 `_classify_event_type` 方法
