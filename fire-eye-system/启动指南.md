# 火瞳系统 - 启动指南

## 📋 前置要求

确保已安装：
- Docker Desktop（用于 Neo4j 和 Redis）
- Python 3.8+（用于后端）
- Node.js 16+（用于前端）

---

## 🚀 启动步骤

### 第一步：启动全部服务（Docker 开发模式）

打开 PowerShell 或 CMD，进入项目目录：

```bash
cd fire-eye-system
docker compose up -d
```

也可以直接双击或执行：

```bash
start-dev.bat
```

开发模式说明：
- `backend` 现在直接挂载本地 `./backend` 目录，并使用 `backend/Dockerfile.dev` 启动，修改 Python 代码后会由 `uvicorn --reload` 自动重载
- `frontend` 现在直接挂载本地 `./frontend` 目录，并使用 `frontend/Dockerfile.dev` 以 `next dev` 模式启动，修改前端代码后会自动热更新
- 日常改业务代码时，不再需要每次手动 `docker compose up --build`
- 只有在修改 `Dockerfile`、依赖文件（如 `requirements.txt`、`package.json`）或基础镜像后，才需要重新 build

生产/演示模式说明：
- 项目新增了独立的 `docker-compose.prod.yml`
- 该文件使用 `backend/Dockerfile.prod` 和 `frontend/Dockerfile.prod`
- 生产模式不会挂载本地源码，也不会启用 `develop.watch`
- 更接近最终交付环境，适合比赛演示和上线前验证

后端 Dockerfile 现已拆分为两套：
- `backend/Dockerfile.dev`：开发环境使用，支持 `uvicorn --reload`
- `backend/Dockerfile.prod`：生产/演示环境使用，关闭自动重载

前端 Dockerfile 现已拆分为两套：
- `frontend/Dockerfile.dev`：开发环境使用，启动快，支持热更新
- `frontend/Dockerfile.prod`：生产/演示构建使用，生成 Next standalone 产物

说明：
- 日常开发启动统一使用 `docker compose up -d`
- `docker-compose.yml` 不再直接写 Neo4j 的 `NEO4J_AUTH`
- Neo4j 初始密码只在 `docker-compose.init.yml` 中用于首次初始化或彻底重置
- Neo4j 保留 `restart: unless-stopped`，避免被 Docker Desktop 或外部 stop 动作后直接退出
- `backend` 现在会等待 Neo4j 和 Redis 健康检查通过后再启动，避免数据库尚未就绪时后端先失败
- 如果之前已经创建过旧容器，首次切换到新的开发模式时需要先执行一次 `docker compose down` 再重新 `up`
- 仅在 Docker Desktop 中点开关不会刷新 compose 配置，也不会重新挂载本地源码

验证服务是否启动：
```bash
docker ps
```

你应该看到至少这些容器在运行：
- `fire-eye-neo4j`
- `fire-eye-redis`
- `fire-eye-backend`
- `fire-eye-frontend`

如果 AstrBot 也需要联调，还应看到：
- `astrbot`

---

### 第二步：确认服务状态

检查后端健康状态：
```bash
curl http://localhost:8000/health
```

或直接在浏览器打开：
- http://localhost:8000/health

应该看到类似：
```json
{"status":"healthy",...}
```

如果后端刚启动时连接不到 Neo4j，请先等待 Neo4j 变为 healthy，再执行：
```bash
docker compose restart backend
```

现在默认情况下，`docker compose up -d` 会先等待 Neo4j 与 Redis 通过健康检查，再启动后端；只有在依赖服务后续异常退出时，才需要手动重启后端。

## 🛑 停止服务

### 停止 Docker 服务
```bash
cd fire-eye-system
docker compose down
```

说明：
- `docker compose down` 会移除容器，但不会删除 Neo4j 数据卷
- 下次 `docker compose up -d` 时会复用已有数据库数据

## 🔁 开关容器推荐命令表

优先使用 `docker compose` 管理整套服务，不建议把 Docker Desktop 中单个容器的开关按钮当作日常主操作方式。

### 1. 日常启动整套服务

```bash
cd fire-eye-system
docker compose up -d
```

适用场景：
- 日常开发启动
- 电脑重启后恢复项目环境
- 一次性启动 Neo4j、Redis、后端、前端、AstrBot

补充说明：
- Compose 会先拉起 Neo4j 和 Redis
- Neo4j 通过健康检查后，后端才会启动
- 后端健康后，前端和 AstrBot 才会继续启动

### 1.1 以生产/演示模式启动整套服务

```bash
cd fire-eye-system
docker compose -f docker-compose.prod.yml up -d --build
```

也可以直接双击或执行：

```bash
rebuild-prod.bat
```

适用场景：
- 比赛演示前做最终环境确认
- 验证生产镜像是否可独立运行
- 避免本地源码挂载影响运行结果

补充说明：
- 该文件是完整的生产 compose 文件，不依赖开发版 compose
- 该模式下前后端都使用 `*.prod` Dockerfile
- 该模式不会继承开发模式的源码挂载和自动热更新配置

### 2. 日常关闭整套服务

```bash
cd fire-eye-system
docker compose down
```

### 2.1 推荐脚本命令

项目根目录已提供 4 个常用脚本：

- `fire-eye-system/start-dev.bat`：启动开发环境
- `fire-eye-system/rebuild-dev.bat`：重建开发环境
- `fire-eye-system/start-prod.bat`：启动生产/演示环境
- `fire-eye-system/rebuild-prod.bat`：重建生产/演示环境
- `fire-eye-system/check-env.bat`：执行环境自检

推荐用法：
- 日常开发启动：`start-dev.bat`
- 修改依赖后开发重建：`rebuild-dev.bat`
- 演示前生产启动：`start-prod.bat`
- 演示前生产重建：`rebuild-prod.bat`
- 演示前环境检查：`check-env.bat`

`check-env.bat` 会检查：
- Docker Desktop / Docker Engine 是否可用
- 是否检测到 `OPENAI_API_KEY`
- 端口 `5000`、`8000`、`7474`、`7687`、`6125` 的监听状态
- `fire-eye-backend`、`fire-eye-frontend`、`fire-eye-neo4j`、`fire-eye-redis` 是否在运行
- 上述核心容器的健康状态是否正常

适用场景：
- 结束当天开发
- 临时关闭整个项目
- 需要按最新 compose 配置整体重建容器

### 3. 只启动 Neo4j

```bash
cd fire-eye-system
docker compose up -d neo4j
```

适用场景：
- 单独排查数据库状态
- 先启动数据库，再等待后端连接

### 4. 只停止 Neo4j

```bash
cd fire-eye-system
docker compose stop neo4j
```

适用场景：
- 临时停止图数据库
- 排查 Neo4j 日志或重启前准备

### 5. 只重启后端

```bash
cd fire-eye-system
docker compose restart backend
```

适用场景：
- Neo4j 已经启动，但后端启动时连接失败
- 修改了环境变量后需要快速重启

补充说明：
- 仅修改后端 Python 代码时，正常不需要手动重启，`uvicorn --reload` 会自动生效
- 如果自动重载未触发，再执行 `docker compose restart backend`

### 6. 依赖或镜像变更后重建服务

```bash
cd fire-eye-system
docker compose up -d --build backend frontend
```

适用场景：
- 修改了 `backend/requirements.txt`
- 修改了 `frontend/package.json`
- 修改了 `backend/Dockerfile.dev`、`backend/Dockerfile.prod`、`frontend/Dockerfile.dev` 或 `frontend/Dockerfile.prod`
- 怀疑容器依赖环境与本地代码不同步

补充说明：
- 当前 `docker-compose.yml` 默认使用 `backend/Dockerfile.dev`
- 当前 `docker-compose.yml` 默认使用 `frontend/Dockerfile.dev`
- 如果要做生产镜像或演示打包，应显式使用：

```bash
docker compose -f docker-compose.prod.yml up -d --build
```

### 7. 强制按最新配置重建 Neo4j 容器

```bash
cd fire-eye-system
docker compose up -d --force-recreate neo4j
```

适用场景：
- 修改了 `docker-compose.yml` 中 Neo4j 配置
- 需要确保旧容器实例被新配置替换

### 8. 彻底重置 Neo4j（会删除数据卷）

```bash
cd fire-eye-system
docker compose down -v
docker compose -f docker-compose.yml -f docker-compose.init.yml up -d
```

适用场景：
- 明确要清空 Neo4j 所有数据
- Neo4j 因旧数据卷和首次初始化状态冲突而异常

注意：
- 这会删除 Neo4j 持久化数据
- 重新初始化后使用初始账号密码 `neo4j/password`

### 不推荐的方式

- 不建议日常通过 Docker Desktop 单独反复点击 `fire-eye-neo4j` 的开关按钮
- 不建议把 `docker compose down -v` 当成普通关闭命令
- 不建议在旧数据卷仍保留时，试图通过修改 `NEO4J_AUTH` 来改密码

## ⚠️ 常见问题

### 问题1：端口被占用
本项目 Docker 默认端口为：
- 前端：5000
- 后端：8000
- Neo4j：7474 / 7687
- AstrBot：6125

如果端口冲突，请修改 `docker-compose.yml` 后重新执行：
```bash
docker compose up -d --build
```

### 问题2：浏览器显示旧样式
- 按 `Ctrl + Shift + R` 强制刷新
- 或使用无痕模式打开

### 问题3：后端连接失败
检查 Docker 容器是否运行：
```bash
docker ps
```

查看后端日志：
```bash
docker logs fire-eye-backend --tail 200
```

如果是 Neo4j 未就绪导致失败，执行：
```bash
docker compose restart backend
```

如果你刚修改了后端代码，但前端仍表现为旧逻辑：
- 先确认当前是按开发模式启动的容器，而不是旧容器
- 再查看后端日志确认是否触发自动重载：

```bash
docker logs fire-eye-backend --tail 200
```

- 如果修改的是依赖或 Dockerfile，不是普通源码修改，需要执行：

```bash
docker compose up -d --build backend
```

### 问题4：前端容器异常
查看前端日志：
```bash
docker logs fire-eye-frontend --tail 200
```

如果前端页面仍显示旧代码：
- 先按 `Ctrl + Shift + R` 强制刷新
- 再确认前端容器是否以开发模式运行
- 如果修改了 `package.json`、`frontend/Dockerfile.dev`、`frontend/Dockerfile.prod` 或 Next.js 构建相关配置，需要执行：

```bash
docker compose up -d --build frontend
```

### 问题5：Neo4j 容器启动后反复重启
如果 `fire-eye-neo4j` 容器不断重启，并且日志里出现类似内容：

```text
Changed password for user 'neo4j'. IMPORTANT: this change will only take effect if performed before the database is started for the first time.
Neo4j is already running (pid:7).
```

通常说明：
- 旧版 Neo4j 容器在创建时带上了 `NEO4J_AUTH=neo4j/password`
- 但 `neo4j_data` 卷里已经有历史数据库状态，`NEO4J_AUTH` 对旧卷不会重新设置密码
- 你在 Docker Desktop 中手动点开关时，Docker 只是重新启动这个旧容器，旧环境变量依然存在
- 旧配置中的自动重启策略也可能让手动 stop/start 过程出现额外重启痕迹

本项目现在采用两套流程：
- **日常启动**：`docker compose up -d`
- **首次初始化或彻底重置 Neo4j**：`docker compose -f docker-compose.yml -f docker-compose.init.yml up -d`

如果你已经改过配置，但 Neo4j 还是在手动点击开关后复现，需要先重建一次旧容器：

```bash
cd fire-eye-system
docker compose down
docker compose up -d
```

如果仍然异常，或者你就是要彻底重置 Neo4j：

```bash
cd fire-eye-system
docker compose down -v
docker compose -f docker-compose.yml -f docker-compose.init.yml up -d
docker logs -f fire-eye-neo4j
```

说明：
- 第一组命令用于重建旧容器，清掉旧容器配置里的 `NEO4J_AUTH`
- 第二组命令用于删除 Neo4j 数据卷并按初始密码 `neo4j/password` 重新初始化
- Neo4j 保留自动重启策略，若被外部 stop 动作中断，Docker 会自动尝试恢复容器
- 以后日常启停请优先使用 `docker compose up -d` / `docker compose down`，不要依赖 Docker Desktop 单独启动旧容器做首次排障

## 🎯 开发建议

1. **统一使用 Docker 管理服务**：
   - 不再使用本地 `start-backend.bat` 启动后端
   - 不再混用宿主机 Python 进程和容器内后端
   - 所有服务统一通过 `docker compose` 管理
   - 日常启动使用 `docker compose up -d`
   - 仅在 Neo4j 首次初始化或彻底重置时使用 `docker-compose.init.yml`
   - Neo4j 保留 `restart: unless-stopped`，避免被意外停止后直接退出

2. **修改代码后**：
   - 如需重新构建镜像：`docker compose up -d --build`
   - 如仅需重启后端：`docker compose restart backend`
   - 前端页面修改后若未同步生效，可重启前端容器

3. **查看日志**：
   - 后端日志：`docker logs fire-eye-backend -f`
   - 前端日志：`docker logs fire-eye-frontend -f`
   - Neo4j 日志：`docker logs fire-eye-neo4j -f`
