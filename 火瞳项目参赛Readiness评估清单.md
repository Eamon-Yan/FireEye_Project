# 火瞳项目参赛 Readiness 评估清单

## 一、总体结论分级
先给你一个简单分级标准，方便团队内部判断：

### A. 可直接答辩 / 可现场演示
满足条件：
- 核心闭环稳定
- 演示流程固定
- 材料齐全
- 现场网络/接口异常也有备选方案

### B. 可参赛提交，但不建议直接现场裸演
满足条件：
- 主体功能已完成
- 有一定测试
- 但仍存在配置不统一、流程未固化、临场风险较高的问题

### C. 暂不建议提交
满足条件：
- 功能点主要停留在模块级
- 关键链路未跑通
- 无法稳定复现结果

**按我目前看到的仓库状态，我会暂时把这个项目评为：B+，接近 A。**
也就是：**已经具备参赛基础，但在“现场稳定答辩”层面还建议再做一轮收尾。**

# 二、核心功能 readiness 清单

## 1. 核心闭环是否真实跑通
这是最重要的一项。

### 必须全部满足
- [ ] 能上传一份真实火灾调查报告 / 示例文档
- [ ] 系统能成功解析文档内容
- [ ] LLM 能输出结构化抽取结果
- [ ] 抽取结果能写入 Neo4j
- [ ] 前端图谱页面能稳定显示节点和边
- [ ] 能基于图谱执行至少一种预测/推演功能
- [ ] 能把最终结果以“评委看得懂”的形式展示出来

### 达标标准
- 至少用 **2~3 份不同样例文档** 跑通完整链路
- 每份样例都能稳定得到可展示结果
- 不依赖人工进数据库手工修数据

### 风险提示
如果目前只能做到：
- 上传成功，但图谱偶尔不显示
- 抽取结果出来，但入库不稳定
- 前端偶发空白、数据错位

那就还不算真正 ready。

---

## 2. 功能边界是否清晰
比赛展示时，功能多不一定更好，**清晰更重要**。

### 核心功能（必须保）
- [ ] 文档上传
- [ ] 火灾事件/因果关系抽取
- [ ] 图谱构建
- [ ] 图谱可视化
- [ ] 场景预测/路径推演

### 加分功能（可选展示）
- [ ] Telegram 插件接入
- [ ] QQ / AstrBot 插件接入
- [ ] 会话管理
- [ ] 统计查询
- [ ] 多平台消息入口

### 判断标准
- [ ] 团队内部能用一句话讲清“核心能力是什么”
- [ ] 团队内部能区分“必须演示”和“可选加分”
- [ ] 即使插件挂掉，主系统演示仍然完整

**建议：答辩时把插件定位成“扩展能力”，不要把它当唯一主线。**

# 三、技术实现 readiness 清单

## 3. 系统是否能稳定启动
### 必须检查
- [ ] `docker-compose up -d` 可以启动主要服务
- [ ] 前端可访问
- [ ] 后端可访问
- [ ] Neo4j 可访问
- [ ] Redis 可访问
- [ ] 环境变量配置完整
- [ ] `.env.example` 与实际运行配置一致

### 达标标准
- 新机器或新环境下，按文档操作可在 **30 分钟内完成部署**
- 不需要靠“记忆里的临时修复命令”才能跑起来

### 风险信号
如果存在以下情况，说明还没完全 ready：
- [ ] 只有某个同学的电脑能跑
- [ ] 启动需要手动进容器修文件
- [ ] 文档写的是 A，实际运行靠的是 B
- [ ] 模型配置、API 地址、端口说明前后不一致

---

## 4. 配置与代码是否收口
### 重点检查
- [ ] 调试代码已清理
- [ ] 临时日志/补丁脚本不会影响正式演示
- [ ] 不存在明显重复或冲突配置
- [ ] 生产/演示配置与开发配置有区分
- [ ] README、部署文档、环境变量模板内容一致

### 你这个项目特别要关注
从现有代码看，建议重点核查：
- [ ] CORS 配置是否重复添加
- [ ] `allow_origins="*"` 与凭证配置是否合理
- [ ] DeepSeek / OpenAI 兼容接口的表述是否统一
- [ ] 前端 API 地址是否与 Docker 运行方式一致
- [ ] 插件配置文件和主系统配置文件是否一致

### 达标标准
- 评委看到的是“一个整理过的工程”
- 不是“很多修复痕迹堆在一起的开发现场”

---

## 5. 测试是否覆盖关键链路
### 至少应具备的测试
- [ ] 后端健康检查通过
- [ ] 上传接口测试通过
- [ ] 查询接口测试通过
- [ ] 图谱查询接口测试通过
- [ ] 前后端联调通过
- [ ] 至少一次完整业务链路测试通过

### 最好补充
- [ ] 异常文件上传测试
- [ ] 空数据库场景测试
- [ ] LLM 返回异常时的兜底测试
- [ ] Neo4j / Redis 连接失败时的提示测试
- [ ] 页面空状态展示测试

### 达标标准
- 不要求测试覆盖率特别高
- 但**核心演示链路必须有可复现测试记录**

# 四、比赛展示 readiness 清单

## 6. 演示链路是否“评委友好”
### 必须准备一条固定 demo 主线
建议按这个顺序：
- [ ] 1 分钟讲问题背景
- [ ] 1 分钟讲系统架构
- [ ] 3 分钟演示上传→抽取→图谱→预测
- [ ] 1 分钟讲创新点
- [ ] 1 分钟讲落地价值

### 演示时必须做到
- [ ] 页面路径固定
- [ ] 样例文件固定
- [ ] 操作步骤固定
- [ ] 结果页面固定
- [ ] 每一步都有讲解词

### 坚决避免
- [ ] 临场翻文件夹找样例
- [ ] 临场改配置
- [ ] 临场重启三四个服务
- [ ] 临场解释“这个功能其实本来可以”

---

## 7. 是否准备了“演示数据包”
### 建议必须有
- [ ] 1 份最稳的标准样例
- [ ] 1 份补充样例（展示泛化能力）
- [ ] 1 份已跑好的图谱结果截图
- [ ] 1 份预测结果截图
- [ ] 1 份录屏备份

### 达标标准
即使现场网络或模型接口抽风，也能继续展示：
- [ ] 预置数据库数据
- [ ] 预加载图谱页面
- [ ] 演示视频可秒切换

**比赛中“有备份”不是减分项，是成熟度加分项。**

---

## 8. 答辩材料是否完整
### PPT 至少应包含
- [ ] 项目背景与痛点
- [ ] 目标用户 / 应用场景
- [ ] 系统总体架构图
- [ ] 核心流程图
- [ ] 算法/模型流程说明
- [ ] 系统功能展示
- [ ] 创新点总结
- [ ] 效果与价值
- [ ] 不足与未来工作

### 最好补充
- [ ] 与传统方案对比
- [ ] 与纯关键词检索方案对比
- [ ] 与普通知识图谱方案对比
- [ ] 你们方案为什么更适合火灾调查场景

### 达标标准
- 评委在不看代码的情况下，也能理解项目价值
- 不是只有技术，没有问题驱动和应用意义

# 五、创新性 readiness 清单

## 9. 创新点是否能说清楚
比赛很看重这一点。

### 至少回答清楚这几个问题
- [ ] 你们解决的核心问题是什么？
- [ ] 为什么不能只靠人工分析？
- [ ] 为什么不能只靠普通检索？
- [ ] 为什么要用“事理图谱”而不是一般知识库？
- [ ] LLM 在这里具体发挥了什么作用？
- [ ] 图谱预测相比静态展示有什么价值？

### 可提炼的创新方向
你们项目大概率可以从这些角度提炼：
- [ ] 火灾调查文本到事理图谱的自动抽取
- [ ] 面向火灾因果链的结构化建模
- [ ] 基于图谱的场景演化预测
- [ ] Web 系统 + Bot 插件的多入口交互
- [ ] 面向应急决策支持的应用落地

### 达标标准
- 创新点不是“用了大模型”
- 而是“**用大模型解决了什么原本难解决的问题**”

# 六、工程质量 readiness 清单

## 10. 工程是否足够像“成品”
### 建议检查
- [ ] 项目目录结构清晰
- [ ] 核心模块命名统一
- [ ] 文档命名和版本状态清晰
- [ ] 无过多“fix_xxx.py / final_fix / really_final”风格文件暴露给评委
- [ ] 非必要的临时文档可归档或隐藏

### 为什么重要
你这个仓库现在从文件名上看，存在不少：
- `fix_*`
- `*_FINAL_*`
- `*_STATUS_*`
- `*_GUIDE_*`

这说明开发过程很真实，但对比赛评委来说，容易显得：
- 工程尚未完全收束
- 状态文档很多但版本不统一

### 达标标准
- 展示版本仓库/压缩包最好是整理过的
- 让评委看到的是主干成果，而不是全部修复历史

---

## 11. 安全与合规是否有基本说明
### 至少应说明
- [ ] 文档上传数据如何处理
- [ ] 是否保存原始文本
- [ ] API Key 是否不会泄露
- [ ] 是否避免把真实敏感事故数据直接公开展示
- [ ] 示例数据是否可合法用于比赛展示

### 达标标准
- 不需要上升到企业级安全
- 但至少要有基本意识和说法

- [ ] 为什么选 Neo4j？

- [ ] LLM 的提示词/抽取策略是什么？
